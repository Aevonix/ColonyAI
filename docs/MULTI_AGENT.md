# Colony Multi-Agent Architecture

Colony's multi-agent support (introduced in v0.7.0) enables multiple agent hosts, coding agents, and other AI systems to connect to a central Colony instance and share unified context.

## Overview

┌─────────────────────────────────────────────────────────────────┐
│                         Colony (Central)                         │
│                                                                  │
│  ┌─────────────┐  ┌──────────────┐  ┌────────────────────────┐  │
│  │ Agent Store │  │ Initiative   │  │ Assignment Engine      │  │

│  └─────────────┘  └──────────────┘  └────────────────────────┘  │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │                    WebSocket Manager                     │    │
│  └─────────────────────────────────────────────────────────┘    │
│                              ▲                                   │

│              ┌───────────────┼───────────────┐                  │

## Key Concepts

### Agents

An **agent** is any system that connects to Colony to receive work or provide capabilities:

- **Local agents**: Connected via HTTP (same trusted network)
- **Remote agents**: Connected via WebSocket (works behind NAT)

### Initiatives

An **initiative** is a unit of work that Colony wants an agent to perform:

- Generated by autonomy loop from blocked goals, neglected contacts, etc.
- Assigned to best available agent based on capabilities, load, priority
- Tracked through full lifecycle: pending → assigned → acknowledged → completed/failed

### Capabilities

Capabilities define what an agent can do:

- `messaging` — Can send/receive messages
- `calendar` — Can manage calendar events
- `coding` — Can write/modify code
- `research` — Can perform web research

## Quick Start

### 1. Invite a Remote Agent

On your Colony host:

```bash
colony agent invite --capabilities messaging,calendar --primary
```

Output:
```
Setup code: COLONY-7X9K-M2P4-QR8W
Expires at: 2026-04-25T18:00:00Z

Run on remote agent:
  colony agent connect --setup-code COLONY-7X9K-M2P4-QR8W --colony-url https://colony.example.com
```

### 2. Connect from Remote Agent

On the remote machine:

```bash
colony agent connect --setup-code COLONY-7X9K-M2P4-QR8W --colony-url https://colony.example.com
```

Output:
```
Agent connected: agent-abc123
Node ID: node-xyz789
Colony ID: colony-1
WebSocket URL: wss://colony.example.com/v1/host/agents/agent-abc123/stream

Config saved to: ~/.colony/agent.json
```

### 3. Run an Agent

Colony supports several agent paths out of the box:

- **Hermes plugin** (`plugins/hermes-plugin/`) — mounts Colony into a Hermes
  agent host. Install with `plugins/hermes-plugin/install.sh`; the poller and
  queue-worker scripts (or the unified `colony-agent-bridge` daemon) forward
  initiatives and `agent_action` jobs to the agent's webhook.
- **MCP** (`colony mcp setup`) — configures a coding harness (Claude Code,
  Codex, Crush, OpenCode) to use Colony over the Model Context Protocol
  (stdio transport).
- **colony-worker daemon** (`colony_sidecar/workers/colony_worker.py`,
  console script `colony-worker`) — an installable, capability-typed worker
  that claims queued jobs and executes them with an LLM in a read/analyse
  posture. Stdlib-only, so it runs on hosts without the full sidecar stack.
- **Python Agent SDK** (`colony_sidecar.agent.AgentClient`) — for custom
  agents; reads `~/.colony/agent.json` and connects via WebSocket (see below).

## Architecture Details

### Agent Registration

#### Local Agents

Local agents register via HTTP POST:

```bash
POST /v1/host/agents/register
{
  "name": "node-a-hermes",
  "connection_mode": "local",
  "gateway_url": "http://192.0.2.10:18789",
  "capabilities": ["messaging", "calendar"],
  "is_primary": true
}
```

#### Remote Agents

Remote agents connect via WebSocket with a setup code:

1. Generate setup code on Colony host: `colony agent invite`
2. Connect from remote: `colony agent connect --setup-code XXX`
3. WebSocket connection established with authentication

### Initiative Flow

```
┌─────────────┐
│ Autonomy    │
│ Loop        │
│ (generates) │
└─────┬───────┘
      │
      ▼
┌─────────────┐
│ Initiative  │
│ Engine      │
│ (creates)   │
└─────┬───────┘
      │
      ▼
┌─────────────┐     ┌─────────────┐
│ Assignment  │────▶│ Agent Store │
│ Engine      │     │ (select)    │
└─────┬───────┘     └─────────────┘
      │
      ▼
┌─────────────┐
│ WebSocket   │
│ Manager     │
│ (delivers)  │
└─────┬───────┘
      │
      ▼
┌─────────────┐
│ Agent       │
│ (executes)  │
└─────────────┘
```

### Assignment Algorithm

The assignment engine scores agents based on:

1. **Capability match** (required)
2. **Status** (must be online/busy)
3. **Load** (current_assignments / max_concurrent)
4. **Priority** (0=backup, 1=normal, 2=high)
5. **Primary status** (primary agents preferred)

Scoring formula:
```
score = 100 * is_primary + 10 * priority + (1 - load) * 100
```

### WebSocket Protocol

#### Authentication

After WebSocket connection, agent sends:

```json
{
  "type": "auth",
  "agent_id": "agent-abc123",
  "node_id": "node-xyz789",
  "signature": "cert-signature"
}
```

Colony responds with:

```json
{
  "type": "auth_ok",
  "seq": 1
}
```

#### Initiative Delivery

Colony pushes initiative:

```json
{
  "type": "initiative",
  "seq": 2,
  "initiative": {
    "id": "init-123",
    "type": "notification",
    "description": "Remind the owner about meeting",
    "priority": 0.8,
    "assigned_at": "2026-04-25T15:00:00Z"
  }
}
```

Agent acknowledges:

```json
{
  "type": "initiative_ack",
  "initiative_id": "init-123",
  "seq": 3
}
```

#### Heartbeat

Agent sends heartbeat every 30 seconds:

```json
{
  "type": "heartbeat",
  "status": "online",
  "current_assignments": 2,
  "seq": 4
}
```

## API Reference

All endpoints are under the `/v1/host` prefix.

### Agent Endpoints

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/agents/invite` | POST | Generate setup code for remote agent |
| `/agents/connect` | POST | Connect remote agent with setup code |
| `/agents/register` | POST | Register local agent |
| `/agents` | GET | List all agents |
| `/agents/{id}` | GET | Get agent details |
| `/agents/{id}` | PATCH | Update agent config |
| `/agents/{id}` | DELETE | Revoke agent |
| `/agents/{id}/heartbeat` | POST | Update agent status |
| `/agents/health` | GET | Get health of all agents |
| `/agents/{id}/stream` | WS | WebSocket for remote agents |

### Initiative Endpoints

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/initiatives` | POST | Create initiative |
| `/initiatives` | GET | List initiatives |
| `/initiatives/{id}` | GET | Get initiative details |
| `/initiatives/{id}/claim` | POST | Claim initiative |
| `/initiatives/{id}/complete` | POST | Mark as completed |
| `/initiatives/{id}/fail` | POST | Mark as failed |
| `/initiatives/{id}` | DELETE | Cancel initiative |
| `/initiatives/{id}/delegate` | POST | Delegate to another agent |
| `/initiatives/{id}/retry` | POST | Retry a failed initiative |
| `/initiatives/{id}/priority` | PATCH | Adjust priority |

## CLI Reference

### Agent Commands

```bash
# Invite remote agent
colony agent invite [--expires 900] [--capabilities messaging,calendar] [--primary]

# Connect as remote agent
colony agent connect --setup-code COLONY-XXXX-XXXX-XXXX --colony-url https://...

# List agents
colony agent list [--status online]

# Show agent details
colony agent show <agent_id>

# Revoke agent
colony agent revoke <agent_id>

# Disconnect this agent
colony agent disconnect
```

### Initiative Commands

```bash
# List initiatives
colony initiative list [--status pending] [--agent <id>]

# Show initiative details
colony initiative show <id>

# Cancel initiative
colony initiative cancel <id> [--reason "reason"]
```

## Agent SDK

### Python

```python
from colony_sidecar.agent import AgentClient

# Load config from ~/.colony/agent.json
client = AgentClient()

# Set initiative handler
@client.on_initiative
async def handle_initiative(initiative):
    print(f"Received: {initiative['description']}")

    # Acknowledge receipt
    await client.acknowledge(initiative["id"])

    # Do work
    result = await process(initiative)

    # Report completion
    await client.complete(initiative["id"], result=result)

# Connect and start
await client.start()
```

The client handles authentication, the heartbeat loop, initiative delivery,
and reconnection with exponential backoff.

### Hermes plugin and workers

For Hermes hosts, the plugin plus the `colony-agent-bridge` daemon (or the
individual poller/queue-worker cron scripts) cover the same circuit without
custom code: initiatives are forwarded to the agent's webhook, `agent_action`
jobs are claimed from the task queue, and the skill index is synced back to
Colony. See `plugins/hermes-plugin/poller/README.md`.

## Security Model

### Setup Codes

- Generated with cryptographically secure random
- Rate-limited (5 failed attempts → 15 min lockout)
- Single-use by default
- Expire after 15 minutes by default

### Node Certificates

- Colony signs node certificates with its private key
- Private key never leaves Colony host
- Certificates include: colony_id, node_id, public_key, signature, expiry

### Authentication Flow

1. Agent generates keypair
2. Agent sends public key during `colony agent connect`
3. Colony signs certificate
4. Agent includes signature in WebSocket auth

## Troubleshooting

### Agent Not Connecting

1. Check setup code hasn't expired
2. Verify Colony URL is reachable
3. Check firewall allows WebSocket connections
4. Run `colony doctor` for diagnostics

### Initiatives Not Being Assigned

1. Check agent status: `colony agent show <id>`
2. Verify agent has required capabilities
3. Check agent isn't at max capacity
4. Check initiative type isn't in `excluded_types`

### WebSocket Disconnects

1. Check network stability
2. Verify heartbeat is running (30s interval)
3. Check logs for error messages
4. Agent auto-reconnects with exponential backoff
