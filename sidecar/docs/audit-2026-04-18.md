# Colony-Core Deep Audit — 2026-04-18

## Critical Issues

### 1. Context engine uses `/context/assemble` instead of `/context/enriched`
**File:** `src/plugin.ts:696`
**Problem:** The context engine's `assemble()` method calls `contextAssemble()` which only queries memory search. The `/context/enriched` endpoint queries all 16 subsystems in parallel (memory, relationships, style, goals, world model, insights) — but nothing calls it.
**Fix:** Switch to `enrichedContext()` in the context engine factory.

### 2. Events WebSocket never receives events
**File:** `sidecar/colony_sidecar/api/routers/host.py:110,487`
**Problem:** `_event_subscribers` queues are created when WebSocket clients connect, but **nothing ever pushes events into them**. The autonomy loop has an `EventBus` instance but never broadcasts to the subscriber queues. The WebSocket is dead — clients connect but never receive data.
**Fix:** Add a bridge function that pushes EventBus events to `_event_subscribers` queues.

### 3. WebSocket events endpoint doesn't authenticate
**File:** `sidecar/colony_sidecar/api/routers/host.py:483-497`
**Problem:** The plugin sends an auth message (`{ type: "auth", token: apiKey }`) after connecting, but the sidecar's `events_ws` handler never reads it — it just calls `ws.accept()` and waits. No auth check.
**Fix:** Read the first message, validate the API key, then start forwarding events.

## OpenClaw Integration Issues

### 4. `contact_id` always uses `sessionKey` as surrogate
**File:** `src/plugin.ts:704`
**Problem:** OpenClaw's `assemble` params don't carry a person/contact ID. The plugin falls back to `sessionKey` which is often a UUID, not a person identifier. Colony's relationship graph and style adaptation need a real person ID.
**Status:** Known gap, documented in code.

### 5. Context engine `ingest()` is a no-op
**File:** `src/plugin.ts:658`
**Problem:** `ingest()` returns `{ ingested: true }` without doing anything. Colony relies on the `reply_dispatch` hook for turn sync instead. This is intentional but means the context engine doesn't participate in OpenClaw's standard ingestion pipeline.

### 6. Events lifecycle service only logs — doesn't act
**File:** `src/plugin.ts:1851-1910`
**Problem:** When the events WebSocket receives a `proactive_message` event, it only logs it. It doesn't actually deliver the message to the channel. The comment says "wired separately once adapter contracts are corrected."
**Status:** Tracked as follow-up. Proactive delivery from Colony doesn't work yet.

### 7. `llm_output` hook returns void — can't modify output
**File:** `src/plugin.ts:1580`
**Problem:** The `llm_output` hook signature returns `Promise<void>`. OpenClaw's SDK allows void returns for observer hooks, so this is correct — but it means Colony can't filter or modify LLM output through this hook. Safety filtering only happens on `message_sending`.

## Code Quality Issues

### 8. AutonomyLoop creates AnomalyDetector on every property access
**File:** `sidecar/colony_sidecar/autonomy/registry.py:31-33`
**Problem:** `registry.anomalies` creates a new `AnomalyDetector()` every time it's accessed. Should be cached or wired like other subsystems.

### 9. `get_definitions()` on ToolExecutor always returns empty
**File:** `sidecar/colony_sidecar/reasoning/executor.py:63-72`
**Problem:** The comment says "reserved for future use" but the ReasoningLoop never passes tool definitions to the LLM call. This means the LLM can never call Colony-native tools server-side — only host-side tools work.

### 10. `message_received` hook derives sessionKey inconsistently
**File:** `src/plugin.ts:1558-1565`
**Problem:** The hook uses `conversationId ?? accountId:channelId ?? channelId:from` as sessionKey. But the safety hook uses `conversationId ?? channelId:to`. These won't match for the same session, so cached inbound text from `message_received` won't be found by the safety hook.

### 11. HostHealthResponse `notes` always says "not wired" even when subsystems are
**File:** `sidecar/colony_sidecar/api/routers/host.py:170-191`
**Problem:** The health endpoint only checks `_graph` and `_response_gate` and `_reasoning_loop` for notes. All other subsystems (goals, contacts, briefings, etc.) have no health note.
